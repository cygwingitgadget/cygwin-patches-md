This patch updates the implementations of stabilize_sig_stack, setjmp
and longjmp on AArch64. It depends on the initial versions that were
posted earlier by
Thirumalai Nagalingam <thirumalai.nagalingam@multicorewareinc.com>:

```
  https://cygwin.com/pipermail/cygwin-patches/2026q1/014586.html
  https://cygwin.com/pipermail/cygwin-patches/2026q1/014587.html
  https://cygwin.com/pipermail/cygwin-patches/2026q1/014588.html

```

I am sending this in a separate thread since the changes are combined.
I will reply to the threads above with a link to this message.

For this AArch64 implementation, the x86_64 versions were used as a
reference for functional parity. The resulting patch was tested and
confirmed to pass the existing unit tests.

However, while reviewing the x86_64 implementation, I noticed some
differences in how the stacklock variable is handled.
In stabilize_sig_stack, setjmp, and longjmp, the lock variable is
decremented, but in sigdelayed, it is explicitly set to 0.

I would like to clarify the intended semantics:

1. Is there a functional difference between decrementing stacklock
   and setting it to zero?
2. Is stacklock ever used as a reference counter (i.e., can it be
   greater than 1)?
3. I see it being decremented in setjmp and longjmp, but I have
```
   not yet found where it is incremented in that context. Could you
   please point me to where that occurs?

```

Thank you,
Igor Podgainoi
